# Risk Register

| ID | Risk | Probability | Impact | Trigger Signal | Response | Stage | Status |
|---|---|---|---|---|---|---|---|
| R-001 | Java version conflict between `AGENTS.md` Java 21 and stage 0 Java 17. | Medium | High | Spring Boot project initialization requires selecting a JDK. | Confirm before Stage 1; default planning assumption is Java 17 because stage 0 MVP is frozen, but do not initialize until confirmed. | 1 | Open |
| R-002 | JWT/Spring Security conflict: `AGENTS.md` lists JWT while stage 0 explicitly bans formal login/JWT/Spring Security. | Medium | High | Stage 1 dependency selection or auth implementation attempts to add security stack. | Keep V1 simulated `X-User-Id`; record conflict; require user confirmation before adding JWT. | 1 | Open |
| R-003 | Requirement creep from excluded AI features such as embeddings, vector DB, LangChain4j, streaming, or multi-Agent. | Medium | High | Implementation tasks start adding non-MVP dependencies or extra modules. | Enforce Stage 0 excluded scope; put new ideas in later-version list. | All | Open |
| R-004 | LLM provider details are not frozen. | Medium | Medium | Stage 4 needs model name, base URL, timeout, retry behavior, or mock strategy. | Define `LlmClient` interface early; externalize provider config; use mock/test double in tests; confirm provider before live call. | 4 | Open |
| R-005 | AI model invocation cost or latency exceeds demo expectations. | Medium | Medium | Repeated manual tests call live model; health checks or cache misses are frequent. | Do not call LLM from health check; add cache in Stage 5; support mock mode for tests; document live-call cost. | 4-5 | Open |
| R-006 | API key or database/Redis secret leakage. | Low | High | `.env`, config files, screenshots, or logs contain secrets. | Commit `.env.example`; ignore `.env`; keep secrets out of trace/log/frontend; review before commit. | All | Open |
| R-007 | RAG quality is weak because V1 uses MySQL `LIKE` only. | Medium | Medium | Questions with synonyms fail despite related knowledge. | Keep expectations clear; improve seed keywords/content; record vector search as later-version candidate. | 4 | Open |
| R-008 | Knowledge cache invalidation with `agent:answer:*` is inefficient or incorrectly implemented. | Medium | Medium | Knowledge changes leave stale answers, or wildcard deletion is slow. | Encapsulate invalidation in cache service; verify with tests; consider key registry only if needed, not before. | 5 | Open |
| R-009 | Data consistency issues because V1 has no physical foreign keys. | Medium | High | Orphaned messages, references, tickets, or logs appear after failed validation. | Enforce Service-layer existence/ownership checks and transactional boundaries; add integration tests. | 2-6 | Open |
| R-010 | Database transaction accidentally includes long LLM HTTP call. | Medium | High | `@Transactional` wraps Agent method that performs HTTP call. | Split transaction boundaries exactly as documented: save user message, call LLM outside DB transaction, save answer/reference. | 4 | Open |
| R-011 | Frontend/backend DTO and VO fields diverge. | Medium | Medium | Frontend expects fields not returned by backend, or casing differs. | Freeze DTO/VO per module before coding; use API document names; add integration checks during Stage 7. | 2-7 | Open |
| R-012 | API document omits full schemas for some endpoints. | Medium | Medium | Implementation task has to guess validation/defaults/VO fields. | At each module task, derive the minimum schema from database/API examples and record assumptions in decision log. | 2-6 | Open |
| R-013 | Ticket state machine is implemented too loosely. | Medium | Medium | Invalid rollback, missing resolution, or non-admin updates pass. | Centralize status transition validation and test all allowed/denied transitions. | 6 | Open |
| R-014 | Async model log failure affects main Agent response. | Low | Medium | Exceptions from async logging propagate or transaction rollback occurs. | Use `@Async` custom executor; catch/log async errors; verify no rollback. | 6 | Open |
| R-015 | Redis failure blocks Agent answers. | Medium | Medium | Redis outage causes request failure instead of fallback. | Catch Redis exceptions and continue DB/LLM path; verify with simulated failure. | 5 | Open |
| R-016 | Required `INNER JOIN`/`LEFT JOIN` appears in docs but not actual mapper SQL. | Medium | Medium | MyBatis-Plus wrappers replace explicit join requirements. | Use XML mapper for required joins and verify SQL during tests/review. | 2-3 | Open |
| R-017 | Docker deployment becomes too complex for MVP. | Medium | Medium | Additional services, CI/CD, orchestration, or production features are added. | Keep Docker Compose only; no Kubernetes/GitHub Actions in V1. | 8 | Open |
| R-018 | Final project presentation is weak despite working code. | Medium | Medium | No README, screenshots, demo data, or deployment guide by Stage 8. | Reserve Stage 8 for README, screenshots, demo users/data, and deployment instructions. | 8 | Open |
| R-019 | Root `README.md` is absent until the packaging/documentation stage. | Medium | Low | Future task expects README setup guidance before Stage 8. | Create README in Stage 8 or earlier if explicitly requested; keep setup notes in implementation docs until then. | 0-8 | Open |
| R-020 | Business code is started before foundation is verified. | Medium | High | Knowledge/Agent/ticket code appears before backend can compile/start. | Enforce stage gates; Stage 1 must compile/start before Stage 2 begins. | 1-2 | Open |
