# WorkMate AI 企业知识库问答 Agent

> 第一版需求规格说明书（V1.0）

## 1. 项目概述

### 1.1 项目名称

WorkMate AI —— 企业知识库问答 Agent

### 1.2 项目背景

企业内部存在大量制度、开发规范、流程文档和常见问题。员工通常需要通过文档搜索或询问同事才能获取答案，效率较低。

本项目旨在开发一个基于大语言模型（LLM）的企业知识库问答 Agent。Agent 不直接依赖模型自由回答，而是优先查询企业知识库，根据检索结果组织回答，从而提高回答准确性，并降低模型幻觉风险。

### 1.3 项目目标

实现一个能够完成以下工作的企业内部知识助手：

- 根据员工问题检索知识库。
- 调用大模型生成基于知识的回答。
- 保存聊天记录。
- 使用 Redis 缓存热门问题。
- 当知识库无法回答时，引导用户创建人工咨询工单。

### 1.4 项目定位

本项目定位为企业内部知识问答 Agent（MVP）。第一版重点完成完整业务闭环，不追求复杂 AI 能力。

## 2. 第一版项目范围

### 2.1 知识管理

- 知识分类管理。
- 知识新增。
- 知识修改。
- 知识详情查询。
- 知识分页查询。
- 知识关键词搜索。
- 知识启用、停用与逻辑删除。

### 2.2 Agent 问答

- 创建聊天会话。
- 用户提交问题。
- 查询 Redis 缓存。
- 使用 MySQL 关键词检索知识。
- 构造 Prompt。
- 调用大模型 API。
- 返回回答与引用知识。
- 返回 Agent 执行过程摘要。
- 知识不足时禁止自由编造。

### 2.3 会话与消息

- 创建会话。
- 查看历史会话。
- 修改会话标题。
- 逻辑删除会话。
- 保存用户消息与 Agent 消息。
- 查看会话历史消息。

### 2.4 工单

- 普通员工创建工单。
- 普通员工查看自己的工单。
- 管理员查看全部工单。
- 管理员更新工单状态与处理结果。

### 2.5 缓存

- Redis 问答缓存。
- 缓存 TTL。
- 缓存命中标识。
- 知识变化后清理问答缓存。
- Redis 故障时降级为直接查询数据库和调用模型。

### 2.6 模型调用日志

异步记录：

- 模型名称。
- 调用耗时。
- 是否成功。
- 是否命中缓存。
- Token 数量（模型接口提供时）。
- 错误信息。

### 2.7 前端

- Agent 聊天页面。
- 知识管理页面。
- 分类管理页面。
- 工单管理页面。
- 模型调用日志页面或简单统计展示。

### 2.8 部署

- Dockerfile。
- Docker Compose。
- MySQL、Redis、Spring Boot、Vue、Nginx 容器化部署。
- 云服务器部署。
- GitHub 代码托管和 README 展示。

## 3. 用户角色

### 3.1 普通员工（EMPLOYEE）

拥有以下能力：

- 获取当前模拟用户信息。
- 查询有效知识分类和知识。
- 创建、查询、修改、删除自己的会话。
- 向 Agent 提问。
- 查看自己的聊天消息。
- 创建人工工单。
- 查看自己的工单。

### 3.2 管理员（ADMIN）

拥有以下能力：

- 管理知识分类。
- 管理知识内容。
- 查看全部工单。
- 更新工单状态。
- 查看模型调用日志和统计。

### 3.3 第一版身份模拟

第一版不开发登录、JWT 和 Spring Security。

请求头统一使用：

```http
X-User-Id: 1
```

预置用户：

```text
1：普通员工 employee_demo
2：管理员 admin_demo
```

后端必须根据数据库中的用户记录与角色进行校验，不能只相信前端传入的信息。

## 4. 核心业务流程

### 4.1 知识维护

```text
管理员提交知识
→ Controller 接收请求
→ Service 校验分类与参数
→ Mapper 写入 MySQL
→ 清理 Redis 问答缓存
→ 返回结果
```

### 4.2 Agent 问答

```text
用户提问
→ 校验用户与会话
→ 保存用户消息
→ 查询 Redis
→ 缓存命中：保存 Agent 消息并直接返回
→ 缓存未命中：检索 MySQL 知识库
→ 无知识：返回知识不足提示并允许创建工单
→ 有知识：构造 Prompt
→ 调用大模型
→ 保存 Agent 回答和引用知识
→ 更新会话时间
→ 写入 Redis
→ 异步记录模型日志
→ 返回结果
```

### 4.3 工单

```text
Agent 返回 canCreateTicket=true
→ 用户确认创建工单
→ 后端校验会话和问题消息归属
→ 创建 PENDING 工单
→ 管理员处理
→ 更新为 PROCESSING / RESOLVED / CLOSED
```

## 8. 数据来源

第一版使用模拟企业知识数据，推荐分类：

- 人事制度。
- IT 支持。
- 研发规范。
- 项目流程。

目标数据量：

```text
20～30 条知识
```

每条知识建议：

- 标题清晰。
- 3～8 个关键词。
- 正文 200～800 字。
- 删除个人隐私。
- 可从 Obsidian 笔记中提取并改写，但不开发自动导入功能。

## 9. 第一版验收标准

1. 管理员能够维护知识分类和知识。
2. 普通员工能够创建聊天会话并提问。
3. Agent 能够基于知识库回答，而不是自由编造。
4. 相同问题第二次请求能够命中 Redis。
5. 无知识时返回 `canCreateTicket=true`。
6. 会话和消息能够持久化。
7. Agent 回答能够关联引用知识。
8. 模型调用日志能够异步记录。
9. 前后端能够完成完整联调。
10. 项目能够通过 Docker Compose 部署到云服务器。
11. 代码托管于 GitHub，包含 README、截图和部署说明。
12. API Key、数据库密码等敏感配置不得提交到 GitHub。
